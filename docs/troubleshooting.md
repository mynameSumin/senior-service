## 1. longtermcare.or.kr — robots.txt 전체 차단

기획 당시 평가등급의 1차 소스로 가정했던 `longtermcare.or.kr`(노인장기요양보험)은
robots.txt가 모든 User-agent에 대해 `Disallow: /`로 설정되어 있었습니다.

```
User-agent:*
Disallow:/
```

정부 사이트의 명시적 차단이라 우회하지 않고 존중했습니다. 대신 아래와 같이 처리했습니다.

- 같은 평가등급 데이터를 **공공데이터포털(data.go.kr)의 공식 재공개 채널**로 받았습니다(4번 참조).
- `longtermcare.or.kr` 시설 상세페이지는 자동 수집 없이 **외부 링크로만** 연결해, 과제의
  "외부 링크 동작" 요건을 우회 없이 충족했습니다.

## 2. 심평원 요양병원 평가등급(HIRA) — khqa.kr은 막혀 있지만 hira.or.kr 자체 페이지는 열려 있었습니다

처음에는 요양병원 적정성평가의 게시 위치를 `khqa.kr`(병원평가통합포털)로 알고 있었는데, 이
포털의 robots.txt는 `Disallow: /`(루트만 허용)로 전체 차단이라 포기했었습니다.

그런데 같은 평가 데이터를 `hira.or.kr` 메인 사이트의 "병원·약국찾기 > 우리지역 좋은병원 찾기"
화면에서도 보여주고 있었고, **이 경로는 robots.txt에서 `/co/`만 막혀 있어 차단 대상이
아니었습니다**(`/ra/hosp/*`는 허용). 화면이 쓰는 JS 파일(`HospitalMap.js`)을 읽어 내부 AJAX
엔드포인트를 역추적했습니다.

```
POST https://www.hira.or.kr/ra/hosp/selectHospSrchPlcHospListAjax.do
body: evlCd=14&goodshwSbjtCd=요양병원&pageIndex=N
```

서비스키 없이 페이지당 20건씩 응답을 주었고, 전국 1,220개 요양병원의 최신 평가등급(1~5등급,
1이 최우수)을 그대로 받을 수 있었습니다. 1~5등급을 서비스 전체에서 쓰는 A~E 스케일로
변환해(`GRADE_1_5_TO_AE`) `evaluations` 테이블에 `source="심평원"`으로
적재했습니다(`crawlers/hira/run.ts`). 응답에 평가연도가 없어(최신 등급만 제공) `eval_year`는
크롤링 시점의 연도로 기록했습니다 — 등급 추이 비교는 이 소스만으로는 계산되지 않습니다.

이 발견 전까지는 "요양병원 평가 정보는 robots.txt로 막혀 있다"고 안내했었는데, 실제로는
khqa.kr만 막혀 있었을 뿐 hira.or.kr 자체 경로는 열려 있었습니다 — robots.txt는 사이트
단위가 아니라 경로 단위로 정확히 확인해야 한다는 교훈을 얻었습니다.

## 3. 실버케어코리아(silvercarekorea.com) — "JS 렌더링" 추정이 틀렸습니다

기획서에는 "정적 fetch 시 본문 미추출 → JS 렌더링 추정 → Playwright 필요"로 적었으나,
실제로 막힌 원인은 JS 렌더링이 아니라 **`detail.php`/`list.php`가 Referer 헤더 부재 시
빈 페이지("No scrap")만 반환하는 단순 체크**였습니다.

Playwright로 렌더링을 시도해도 동일하게 빈 페이지가 나와 처음엔 "여전히 막혔다"고
오판했습니다. 실제로는 curl에 `Referer: https://www.silvercarekorea.com/silver/list.php?gubun=A`
헤더 하나만 추가하면 완전한 서버사이드 정적 HTML이 그대로 내려옵니다. 결과적으로
Playwright/Chromium 없이 axios + Referer 헤더만으로 충분했고, 크롤링 속도와 안정성 모두
더 좋아졌습니다.

또한 시설 상세페이지의 `schema.org LocalBusiness` JSON-LD에 후기 본문이 raw 줄바꿈 문자를
포함한 채 들어 있어 `JSON.parse`가 그대로는 실패했습니다(`Bad control character in string
literal`). 문자열 리터럴 내부의 줄바꿈만 골라 `\n`으로 이스케이프하는 소규모
sanitizer(`crawlers/silvercarekorea/run.ts`의 `sanitizeJsonLd`)를 작성해 해결했습니다.

결과적으로 요양원/주야간보호/방문요양 3개 유형 × 15페이지(유형별 약 150건) = **450건 처리,
366개 고유 시설을 적재**했습니다(사이트의 "오늘의 추천" 등 동적 추천 박스가 페이지 경계에서
겹쳐 일부 uid가 중복 처리되었는데, upsert 키가 `external_urls.silvercarekorea` URL이라
자동으로 dedup되었습니다).

## 4. data.go.kr — "Open API 활용신청" 없이도 원본 파일을 받을 수 있었습니다

기획 당시엔 "공식 Open API 서비스키 활용신청 → 발급 대기"가 필요하다고 가정했습니다.
실제로는 데이터셋 상세 페이지(`/data/{pk}/fileData.do`)의 다운로드 버튼이 활용신청과
무관하게 누구나 호출 가능한 2단계 공개 엔드포인트로 동작했습니다.

```
GET /tcs/dss/selectFileDataDownload.do?publicDataPk={pk}
  → { atchFileId, fileDetailSn, status: true } JSON

GET /cmm/cmm/fileDownload.do?atchFileId={id}&fileDetailSn={sn}&dataNm={name}
  → 실제 CSV/XLSX 바이너리
```

이 두 엔드포인트는 `fn_fileDataDownload` 함수(공공데이터포털 공통 JS
`script_cmmFunction.js`)를 프론트엔드 네트워크 흐름 추적으로 역추적해 찾았습니다.
서비스키 발급 대기 없이 즉시 다음 두 데이터셋을 원본째로 받을 수 있었습니다.

- `국민건강보험공단_장기요양기관 평가 결과`(publicDataPk=15104801) — 2019~2024년 다년치
  A~E 등급과 5개 영역별 점수가 그대로 들어 있어, 등급 하락 추이 계산에 그대로 사용했습니다.
- `국민건강보험공단_장기요양기관 시설별 현황`(publicDataPk=15124763) — 시설 일반/입소/인력
  현황입니다.

파일은 CP949(EUC-KR) 인코딩 CSV였고, 시설명에 쉼표가 포함된 행이 있어 단순 `split(",")`이
아니라 `csv-parse`로 제대로 파싱해야 했습니다(`iconv-lite`로 cp949 → utf-8 디코딩 후 파싱).

평가 데이터에는 "인력" 단독 영역 점수가 없습니다(기관운영/환경및안전/수급자권리보장/
급여제공과정/급여제공결과 5개 영역뿐입니다). `lib/scoring.ts`의 인력 가산점 로직은 이
데이터만으로는 항상 0으로 계산됩니다 — 데이터의 한계이며 버그는 아닙니다.

facilities 테이블과의 매칭은 거짓청구 명단공표와 동일한 `bestFacilityMatch`(이름 70% +
주소 30% 가중 유사도) 함수를 재사용해 confidence ≥ 0.6인 행만 evaluations에 적재했습니다.

## 5. 거짓청구 명단공표(보건복지부) — 매칭 0건은 데이터의 한계

`mohw.go.kr/claimList.es?mid=a10507010200`(건강보험 거짓청구 요양기관 명단공표)는 정적
HTML 테이블로 실제 크롤링이 가능했고 44건이 정상 파싱되었습니다. 그러나 이 명단은
"건강보험" 거짓청구 기관(의원/한의원/한방병원/약국 등) 명단이며, 표본 시점 데이터에는
우리 타겟인 요양원/요양병원이 한 건도 포함되지 않아 facilities와의 fuzzy 매칭이
0/44건으로 나왔습니다.

장기요양기관 전용 행정처분 명단은 별도 채널(`longtermcare.or.kr`)에 있을 가능성이
높지만, 해당 사이트는 1번 사유로 robots.txt에 의해 접근하지 않았습니다. 매칭 0건은
크롤러 버그가 아니라 "요양원 전용 행정처분 공개 채널이 일반에 막혀 있다"는 시장 정보
비대칭 그 자체를 보여주는 결과로, 오히려 기획서 1.3절의 가설(부정 정보는 구조적으로
숨겨져 있다)을 뒷받침합니다.

## 6. Next.js Turbopack — 한글 경로에서 패닉

로컬 개발 디렉터리 경로(`/Users/.../모두닥/senior-service`)에 한글이 포함되어 있는데,
Next.js 16의 기본 dev/build 엔진인 Turbopack이 `/results` 페이지(서버 컴포넌트 + RSC
액션 청크)를 빌드할 때 내부 식별자 문자열을 고정 바이트 오프셋(21바이트)에서 자르다가
멀티바이트 UTF-8 문자(`ㅜ`)의 중간을 잘라 panic을 일으켰습니다.

```
panicked at turbopack-core/src/ident.rs:354:34:
start byte index 21 is not a char boundary; it is inside 'ᅮ' (bytes 19..22) of
`Documents_모두닥_senior-service__next-internal_server_app_results_page_actions_...`
```

`/`, `/match`처럼 라우트 경로 문자열이 짧은 페이지는 21바이트 경계를 넘지 않아 우연히
패닉을 피했고, 더 긴 식별자가 생성되는 `/results`에서만 재현되었습니다. 디렉터리명을
바꾸는 대신 `next dev --webpack` / `next build --webpack`으로 빌드 엔진을 webpack으로
전환해 회피했습니다(`package.json`의 `dev`/`build` 스크립트에 반영).

## 7. 시설별 상세조회 Open API — 클릭 시 실시간 호출

`/data/15058856/openapi.do`("국민건강보험공단 장기요양기관 시설별 상세조회 서비스")는
웹 UI의 스웨거 뷰어가 일부 오퍼레이션만 인라인으로 노출하고 나머지는 세션 종속
AJAX(`/tcs/dss/selectApiDetailFunction.do`)로 가려져 있어 막혀 있었습니다. 페이지에
걸린 **공식 활용가이드 docx**(`(활용가이드)장기요양기관 시설정보 수정.docx`, 같은
`/cmm/cmm/fileDownload.do` 패턴으로 다운로드 가능)를 열어 9개 오퍼레이션의 정확한
엔드포인트·요청 파라미터·응답 필드를 전부 확인했습니다.

- `getGeneralSttusDetailInfoItem02` 일반현황, `getStaffSttusDetailInfoItem02` 인력현황,
  `getInsttSttusDetailInfoItem02` 시설현황, `getAceptncNmprDetailInfoItem02` 입소인원,
  `getProgramSttusDetailInfoList02` 프로그램현황, `getNonBenefitSttusDetailInfoList02`
  비급여현황, `getConvInsttDetailInfoList02` 협약기관, `getWlfareToolDetailInfoList02`
  복지용구, `getInsttEtcDetailInfoItem02` 기관기타 — 전부
  `http://apis.data.go.kr/B550928/getLtcInsttDetailInfoService02/{operation}` 형태입니다.

대부분 `longTermAdminSym`(장기요양기관기호) + `adminPttnCd`(기관유형코드)를 요구하는데,
우리 시설은 실버케어코리아발이라 이 공식 코드가 없었습니다. "장기요양기관 시설별
현황" 파일(15124763)의 일반현황·입소인원 시트를 받아 이름+주소 fuzzy 매칭으로 366개 중
363개(99%)에 코드를 채웠습니다(`crawlers/data-go-kr/match-admin-sym.ts`).

이후 `/facility/[id]` 서버 컴포넌트가 매칭된 시설에 대해 인력현황·입소인원·프로그램현황
API를 **요청마다 실시간으로 호출**합니다(`lib/dataGoKrDetail.ts`, `DATA_GO_KR_SERVICE_KEY`는
서버 전용 env로 클라이언트에 노출되지 않습니다). 첫 구현에서는 API 응답 객체를 통째로
순회해 인력 카드를 그렸다가 `longTermAdminSym` 같은 비-인력 필드까지 "14136000904명"으로
잘못 표시되는 버그가 있었습니다 — 화이트리스트(`STAFF_LABELS`) 키만 순회하도록 고쳤습니다.

## 8. 같은 이름+주소로 기관코드가 여러 개 — 더 채워진 코드를 신뢰

실제 시설(엔젤실버요양원)을 클릭해 인력현황을 비교해보니 실버케어코리아/케어닥에 나오는
값과 우리 서비스 값이 달랐습니다. 원인은 같은 이름+주소로 `장기요양기관코드`가 두
개(14313000353, 14313000382) 등록돼 있었던 것이었습니다 — 후기에 "대표가 변경되었다"는
언급이 있어, 운영자 변경으로 코드가 재발급된 것으로 추정됩니다. 새 코드는 아직 현황
신고가 안 됐는지 데이터가 거의 비어 있었고(직접 API로 호출해 공단 쪽 응답 자체가
비어있음을 확인했습니다 — 클라이언트 버그가 아니었습니다), 매칭 로직이 이름+주소만 보고
마지막에 본 코드로 덮어쓰면서 빈 코드가 선택되어 있었습니다.

이름+주소만으로는 어느 코드가 "현재 운영 중인" 코드인지 데이터만으로 100% 단정할 수
없어, 데이터가 더 채워진 코드를 우선하는 정책으로 정했습니다(인력현황 숫자 컬럼 총합 =
"richness", 더 큰 쪽을 대표로 선택). `crawlers/data-go-kr/match-admin-sym.ts`에 적용했고,
이후 전국 시드(9번 항목)에도 동일하게 적용했습니다.

## 9. 전국 시드(517 → 25,262곳)에서 드러난 스케일 문제들

"전국"을 선택해도 실제로는 실버케어코리아가 등록해둔 500여 곳만 보이고 지역별 커버리지가
매우 얇았습니다(세종 2곳, 울산 12곳 등). data.go.kr "시설별 현황" 파일은 일반현황 시트에
전국 30,595행 전수와 기관코드가 이미 들어 있어서(`crawlers/data-go-kr/seed-facilities.ts`),
기존 시설과 fuzzy 매칭되지 않는 나머지를 그대로 채워 25,262곳으로 늘렸습니다. 이 과정에서
실버케어코리아 데이터에는 없던 문제들이 한꺼번에 드러났습니다.

- **PostgREST 기본 1000행 컷**: `.range()` 없이 `select()`하면 1000행에서 조용히
  잘립니다. 517곳일 땐 드러나지 않았던 버그라, 시드 이후 평가등급 매칭 풀과
  `compute-risk-scores.ts`의 대상 시설 목록이 전부 1000개로 잘려 있었습니다.
  `crawlers/shared/db.ts`에 `fetchAllRows()` 페이지네이션 헬퍼를 추가해 모든 전체
  조회 지점에 적용했습니다.
- **O(n×m) fuzzy 매칭 폭발**: 평가등급 CSV(약 8천 매칭 대상 행)를 25,250개 후보 전체와
  매번 전체 비교하면 약 2억 회 문자열 비교가 발생해 10분 넘게 멈춘 것처럼
  보였습니다(`ps`로 CPU 사용률을 보고 죽은 게 아니라 느린 것임을 확인했습니다). 이름을
  정규화해 정확히 일치하는 후보만 먼저 추리는 인덱스(`buildNameIndex`/`matchWithIndex`,
  `crawlers/shared/fuzzy.ts`)를 추가해 8.8초로 줄였습니다 — 단, 철자가 살짝 다른 경우는
  못 잡는 트레이드오프가 있습니다. 두 데이터 모두 같은 공단 등록정보 기반이라 정확히
  일치하는 경우가 대부분이고, 안전하게 건너뛰는 쪽(과매칭보다 미매칭)이 데이터 정합성
  면에서 낫다고 판단했습니다.
- **행 단위 순차 upsert**: `for` 루프 안에서 매 행마다 `await db.upsert(...)`를 부르면
  네트워크 왕복이 수천 번 쌓입니다(평가등급 크롤러, 점수 재계산 스크립트). 배열에 모아서
  500건씩 배치 upsert하도록 바꿨습니다 — 점수 재계산도 N+1 쿼리(시설마다
  evaluations/violations 따로 조회)를 한 번에 전체 조회 후 메모리에서 그룹핑하는 방식으로
  바꿨습니다.
- **이미 코드가 있는 시설을 재매칭해서 덮어쓸 뻔함**: `match-admin-sym.ts`가 풀 크기와
  무관하게 "facilities 전체"를 대상으로 fuzzy 매칭을 돌리고 있었는데, 풀이 25,250개로
  커지자 동명 시설 오매칭 위험이 커졌습니다 — 시드 단계에서 이미 원본 데이터로 직접 채운
  코드를 다시 fuzzy 매칭으로 덮어쓸 뻔했습니다. `.is("long_term_admin_sym", null)` 가드를
  추가해 코드가 없는 시설만 대상으로 하도록 고치고, 의심되는 시점까지의 시드 데이터를
  전부 지우고(evaluations는 cascade로 같이 삭제) 처음부터 다시 정확하게 채웠습니다.

## 10. 요양병원 크롤러가 동명의 요양원에 평가를 잘못 붙였습니다

`crawl:hira`를 처음 돌렸을 때 "신규 시설 1,012곳, 평가등급 적재 1,033건"으로
끝났는데, 1,220건 중 208건이 "기존 시설과 매칭"되어 새로 만들어지지 않았습니다.
이상해서 확인해보니 매칭 대상 풀이 `facilities` 전체(요양원/주야간보호/방문요양 포함
25,262곳)였던 게 문제였습니다 — **요양병원은 건강보험(의료기관) 영역이고 요양원 등은
장기요양보험(요양기관) 영역이라 같은 시설일 수 없는데**, 이름이 비슷한 요양원(예:
"OO요양병원" vs "OO요양원")에 `bestFacilityMatch`가 0.6 이상의 신뢰도로 오매칭한
것이었습니다. 검증해보니 정확히 26건의 평가가 `type="요양원"`인 시설에 잘못 붙어
있었습니다.

같은 기관 종류가 아니면 애초에 매칭 후보가 될 수 없으므로, 매칭 풀을 기존
`type="요양병원"` 시설로만 제한하도록 고쳤습니다(`crawlers/hira/run.ts`) — 재실행 시
중복 생성만 막고, 다른 종류의 시설과는 절대 매칭되지 않습니다. 오매칭된 26건의
evaluations를 지우고 크롤러를 재실행해 해당 26곳을 신규 요양병원 시설로 올바르게
적재했습니다. 일반화하면 **fuzzy 매칭은 후보 풀을 "같은 종류"로 먼저 좁힌 뒤에
돌려야 한다**는 교훈입니다 — 전체 풀 대상 매칭은 후보가 늘어날수록 동명 오매칭
위험이 커집니다(8번 항목의 운영자 변경 케이스와 같은 종류의 문제입니다).

## 11. 요양병원 비급여 비용 — 다른 제공기관 키라 처음엔 Forbidden, 원인은 http/https

요양원 등은 국민건강보험공단 API(제공기관 B550928)로 비급여 비용을 실시간 조회하고
있었는데, 요양병원은 장기요양보험 영역 밖이라 이 API에 데이터가 없었습니다(2번 항목
참고). 같은 평가 데이터를 찾았던 것과 같은 방식으로 data.go.kr에서
"건강보험심사평가원_비급여진료비정보조회서비스"(제공기관 B551182)를 찾았는데, 기존
`DATA_GO_KR_SERVICE_KEY`로 호출하면 `Forbidden`이 떴습니다.

data.go.kr Open API는 **계정 단위가 아니라 제공기관(API)별로 개별 활용신청**이
필요합니다 — B550928용으로 승인된 키라고 B551182도 자동으로 열리지는 않습니다.
사용자가 B551182를 별도로 활용신청·승인받은 뒤 같은 키로 다시 호출했는데도 여전히
`Forbidden`이 떴습니다. `curl -i`로 응답 헤더를 보니 본문이 XML 에러 응답이 아니라
순수 텍스트 `Forbidden`(10바이트)이었던 게 단서였습니다 — data.go.kr 게이트웨이는
`http://`로 들어오는 이 API 요청을 그냥 막아버리고, **`https://`로 호출하니 정상 XML
응답**이 돌아왔습니다(기존 B550928 API는 `http://`로도 잘 동작해서 의심하지
못했습니다). 같은 게이트웨이라도 제공기관마다 프로토콜 강제 여부가 다를 수 있다는
뜻입니다.

해결 후 `lib/hiraNonPayment.ts`를 추가해 평가등급 적재 시 같이 저장해둔 ykiho로 시설
상세페이지에서 실시간 조회하도록 연결했습니다(요양원의 비급여 조회와 동일한 "클릭
시점 실시간 호출" 패턴입니다).

## 12. 원본 페이지 외부 링크 — 실버케어코리아 말고는 비어 있었습니다

과제 제한사항에 "크롤링한 원본 페이지로 이동하는 외부 링크"가 명시돼 있는데,
`external_urls`를 실제로 채우는 건 `crawlers/silvercarekorea/run.ts`뿐이었습니다 —
`seed-facilities.ts`와 `crawlers/hira/run.ts`는 둘 다 `external_urls: {}`로 비워둔
채 적재하고 있었습니다(전체 시설의 대다수가 이 두 크롤러 출처였습니다).

소스별로 실제로 열리는 링크를 다시 확인해야 했습니다.

- **요양병원(HIRA)**: 7번 항목에서 "JSON이 아니라 HTML이라 포기"했던
  `hospInfoAjax.do?ykiho=...`가, 외부 링크 목적으로는 오히려 정답이었습니다 —
  Playwright로 렌더링해보니 그 자체로 병상수·진료과목·인증정보까지 다 보여주는
  완성된 상세페이지였습니다.
- **요양원/주야간보호/방문요양(data.go.kr 시드분)**: longtermcare.or.kr의 시설 상세
  URL 패턴(`selectLtcoSrchDetail.web?ltcAdminSym=...&adminPttnCd=...`)을 찾아
  Playwright로 직접 열어봤지만 탭 껍데기만 뜨고 본문이 비어 있었습니다(세션 기반
  POST 흐름이라 GET 직접 접근이 안 됩니다 — 1번 항목에서 이미 robots.txt로 차단된
  이유와는 별개로, 차단이 없었어도 이 방식으로는 안 됐을 것입니다). 대신 실제로 이
  시설 데이터를 받아온 data.go.kr 데이터셋 페이지(`/data/15124763/fileData.do`)로
  연결했습니다 — "크롤링해온 곳"을 더 정확하게 반영하는 링크입니다(13번 항목에서
  케어닥으로 다시 한 번 개선했습니다).
- **거짓청구**: `violations.source_url`은 처음부터 DB에 저장하고 있었는데
  상세페이지 UI에서 렌더링을 빼먹고 있었습니다. 단순 누락이라 바로 추가했습니다.

이미 적재된 기존 행(빈 `external_urls`로 들어간 요양원 24,944곳, 요양병원 1,033곳)은
두 크롤러를 고친 뒤 `crawlers/data-go-kr/backfill-external-urls.ts` 일회성 스크립트로
채웠습니다.

## 13. 케어닥 — 시설별 실제 페이지가 있으면 그쪽으로, 없으면 data.go.kr로

12번에서 data.go.kr 데이터셋 페이지로 채운 링크는, 모든 시설이 같은 URL을 가리키는
구조라 "다른 서비스의 실제 콘텐츠 페이지"라는 요건엔 약하다는 지적을 받아, 케어닥
(caredoc.kr)에 시설별 실제 페이지가 있는지 확인해보기로 했습니다.

케어닥 시설 페이지는 `/facility/{이름}-LTC-{long_term_admin_sym}-{admin_pttn_cd}`
(요양병원은 `-HSP-{ykiho}`) 형태였고, 우리가 이미 갖고 있는 기관코드로 URL을 직접
구성할 수 있었습니다. 다만 케어닥이 전국 시설을 다 갖고 있는 건 아니라서, 처음 작은
샘플로는 적중률이 낮아 보였습니다. 실제로 17,131곳 전체를 돌려보니 73.6%(12,606곳)가
적중했습니다 — 작은 샘플이 우연히 비주류 기관유형코드에 몰려 있었던 것입니다. 케어닥은
Next.js라 없는 시설도 HTTP 200을 주지만, 서버 렌더링 HTML에 `<title>` 태그 자체가
없다는 점으로 존재 여부를 구분했습니다(`crawlers/caredoc/run.ts`).

## 14. fetchAllRows — ORDER BY 없이 페이지네이션하면 행이 누락됩니다

`crawlers/data-go-kr/backfill-external-urls.ts`를 돌렸을 때 "data.go.kr 링크:
24944곳"으로 끝났는데, 나중에 보니 type="방문요양"인 7,614곳이 여전히
`external_urls`가 빈 채로 남아 있었습니다. 9번 항목에서 추가한 `fetchAllRows()`가
`.range()`로 페이지를 나눠 가져오는데, 호출하는 쿼리에 `.order()`가 없었습니다 —
Postgres/PostgREST는 ORDER BY 없이 OFFSET 페이지네이션을 하면 페이지마다 다른 스캔
순서를 고를 수 있어, 일부 행이 어느 페이지에도 안 걸리고 빠질 수 있습니다(이미 1000행
컷 문제로 한 번 겪었던 것과 같은 계열의 버그인데, 이번엔 컷이 아니라 누락이라 더 찾기
어려웠습니다 — 카운트가 그럴듯하게 맞아 보여서 한동안 알아채지 못했습니다).
`fetchAllRows`를 쓰는 모든 호출부에 `.order("id")`를 추가하고, 누락됐던 7,614곳을
다시 백필했습니다.

## 15. seed-facilities.ts에도 같은 O(n×m) 매칭 폭발이 남아 있었습니다

전국 시드 이후 `npm run crawl:seedfacilities`를 다시 돌렸을 때 15분 넘게 멈춰 있다가
`TypeError: fetch failed` / `write EPIPE`로 실패했습니다. 로그를 보면 매칭 단계
("기존 시설과 중복(건너뜀): 25904곳... 신규 삽입 대상: 109곳")까지는 끝났는데, 그
다음 insert 호출에서 끊긴 것이었습니다.

9번 항목에서 이미 겪었던 것과 똑같은 원인이었습니다 — 이 크롤러는 후보 27,135곳을
`existingPool` 전체(시드 이후 26,000곳 이상으로 커진 상태)와 매번 1:1
비교하는 `bestFacilityMatch`를 그대로 쓰고 있었습니다. 약 7억 회 문자열 비교가
발생해 매칭 단계가 한참 멈춰 있었고, 그 사이 Supabase 커넥션이 유휴 상태로 끊겨
마지막 insert에서 EPIPE가 났습니다. 평가등급 크롤러(`crawlers/data-go-kr/run.ts`)는
그때 `buildNameIndex`/`matchWithIndex`로 고쳤지만, 같은 패턴을 쓰는
`seed-facilities.ts`는 고치지 않고 지나쳤던 것입니다.

같은 인덱스 매칭으로 바꾸자 매칭+삽입 전체가 12초로 끝났습니다(신규 1,231곳 삽입 —
정확히 일치하는 이름만 후보로 추리는 인덱스 방식이라, 기존 fuzzy 검색이 우연히
0.6 이상으로 매칭해 "중복"으로 건너뛰었던 항목 일부가 이번엔 신규로 잡혔습니다).
일반화하면 **같은 버그를 한 곳에서 고쳤다고 끝난 게 아니라, 같은 패턴을 쓰는 다른
파일도 같이 찾아서 고쳐야 한다**는 교훈입니다.
